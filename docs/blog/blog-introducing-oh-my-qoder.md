# 当 AI 编程助手学会「分身术」：oh-my-qoder 多智能体编排实战

> AI 写代码这件事，真正难的从来不是“写出几行代码”，而是“把一整件事做完整”。

这篇文章分四部分：

1. **介绍**：OMQ 想解决什么问题
2. **快速体验**：几分钟跑通一次真实使用
3. **核心设计思路**：它为什么能更稳、更快
4. **总结**：一句话回看 OMQ 的价值

---

## 一、介绍

### 你可能也踩过这些坑

刚开始用 AI 写代码，通常都挺顺手。问题往往出在项目复杂起来之后：

- 说“重构认证模块”，它改了 A 忘了 B，改完 B 又把 C 弄坏；
- 会话一长就丢上下文，下次还得从头解释项目约定；
- 说“完成了”，但没跑测试、没过类型检查；
- 任务一多只能串行，前后端和数据库没法一起推进。

这不是模型不够聪明，而是**缺一层工程化编排**。

### oh-my-qoder 是什么

oh-my-qoder（OMQ）是跑在 Qoder CLI 外层的一套编排插件。它不替代助手，而是把“一个助手”变成“一个分工明确的小团队”。

核心就三件事：**分工、流程、验收**。

```
用户输入 → Hooks（识别事件）→ Skills（注入工作方式）
       → Agents（执行任务）→ State（记录进度）
```

你只需要说清目标，OMQ 负责把任务分发、推进和收尾，最后用证据确认“真的完成”。

## 二、快速体验

### 前置要求

- **Qoder CLI**（`qodercli`）：先安装并登录
- **Node.js >= 20**、**git**
- **tmux**（可选）：需要并行 worker 时再装

### 1）安装

OMQ 是 Qoder CLI 插件，安装方式是“本地目录安装”。所以流程是：先克隆并构建，再把当前目录装进去。

```bash
git clone https://github.com/spring-ai-alibaba/oh-my-qoder.git
cd oh-my-qoder
npm install
npm run build
qodercli plugins install "$(pwd)"
```

然后重启 Qoder CLI，或者在会话里执行 `/plugins reload`。

### 2）初始化和体检

进入 Qoder CLI 会话后执行：

```text
/oh-my-qoder:omq-setup     # 生成 ~/.qoder/AGENTS.md，配置 hooks / HUD / MCP
/oh-my-qoder:omq-doctor    # 检查安装是否正常
```

### 3）一句话启动工作流

日常使用就一个模式：**关键词 + 一句话任务**。

```text
autopilot 构建一个带测试的任务管理 REST API
ralph 修复所有失败测试，不通过不停
ultrawork 给 users 和 orders 接口都加分页
team 3:executor 实现计费模块
plan 为搜索服务设计缓存层
```

### 4）你会看到什么

同样是“帮我写代码”，OMQ 会按任务类型切换执行方式：

```
你：帮我构建一个 REST API
OMQ：[autopilot] 从需求到代码到测试，一路推进到验证通过
```

```
你：同时开发前端、后端和数据库
OMQ：[team 3:executor] 三路并行开发，再做集成验证
```

```
你：彻底重构认证模块
OMQ：[ralph] 循环推进，直到验证通过才停
```

> 你也可以不用“标准关键词”。比如 “build me a…” 往往会触发 autopilot，“keep going” 往往会触发 ralph。只是显式关键词最稳。

## 三、核心设计思路

如果只看表面，OMQ 像是在“调更多 Agent”。但它真正有效的地方，其实是把团队协作里最关键的几件事做成了默认行为：**谁来做、按什么节奏做、做到什么算完成**。

### 1）先把“谁做什么”分清楚

真实团队里，需求分析、方案设计、编码实现、验收测试通常不是一个人全包。OMQ 也是一样：每个角色有明确职责，也有明确边界。

一个常见流程大致是：

```
explore → analyst → planner → critic → executor → verifier
```

可以把它理解成接力：

- `explore` 先把代码地形摸清楚；
- `analyst` 把需求讲明白，补齐隐含约束；
- `planner` 把实施步骤排出来；
- `critic` 专门挑计划漏洞；
- `executor` 动手实现；
- `verifier` 负责最后验收。

重点不在“角色数量”，而在**边界清晰**。比如做计划的人不负责给自己计划打分，做实现的人不负责拍板“已经没问题了”。这能明显减少“看起来做完了，其实漏了一半”的情况。

### 2）同一句需求，用不同推进方式

“帮我做这个”这句话很短，但任务差别可能很大。改个文案和重构一个模块，不应该用同一种节奏。OMQ 把常见工程打法沉淀成几种模式，按场景自动选：

- **Team**：像项目流水线，按 `plan -> prd -> exec -> verify -> fix` 推进。验证不过会回到修复，直到通过或达到上限。
- **Autopilot**：从一句话目标到可交付结果，适合“我只给目标，你帮我推进全流程”。
- **Ralph**：适合复杂重构，核心就是“不验收通过不结束”。
- **Deep Interview**：需求模糊时先问清楚，避免一开始就跑偏。

实战里你记住 `autopilot`、`team`、`ralph` 三个关键词基本就够用了。

### 3）模型路由：不是越贵越好

OMQ 默认会按任务复杂度自动分档：简单任务用轻量模型，常规开发用标准模型，复杂推理再上高阶模型。

这件事听起来像“省钱优化”，但更关键的是**节奏更对**：小任务不会被过度思考拖慢，大任务也不会因为模型太轻而反复返工。你也可以按项目和角色覆盖默认路由。

### 4）把“验收”变成硬规则

很多 AI 编程体验不稳定，核心原因是“完成标准太软”。OMQ 的做法很直接：**没有验证证据，就不算完成**。

它会根据改动规模自动选择验证力度：

| 验证级别 | 适用场景 | 核心检查 |
|---------|---------|---------|
| 轻量 | 小改动（文件少、测试覆盖好） | 类型检查 + lint |
| 标准 | 常规功能开发 | 构建 + 测试 |
| 严格 | 大改动或架构/安全相关 | 全量验证与审查 |

另外有一条很实用的规则：只要改动涉及 `auth`、`secrets`、`.env` 这类敏感区域，验证会自动升级到最严格档。这样可以避免“改动看起来不大，但风险其实很高”的盲区。

### 5）让上下文跨会话“续上”

用 AI 开发最烦的体验之一是：昨天说过的约定，今天还要再说一遍。OMQ 通过两层记忆减少这件事：

- **notepad**：在上下文压缩前先保存关键信息，压缩后再注入回来；
- **project memory**：把项目约定、常用命令、关键决策长期保留，下一次会话自动带上。

效果很直观：你不会每次都重新解释“我们怎么跑测试、这个项目的边界在哪、哪些事情不能做”。

### 6）Hooks：自动化背后的“触发器”

OMQ 的自动能力不是“猜出来的”，而是由 Hook 在关键时刻触发：

- 你提交消息时，关键词检测 Hook 判断是否进入某种模式；
- 会话要结束时，持续模式 Hook 会检查是否满足退出条件；
- 上下文将压缩时，预处理 Hook 会先保存关键信息。

所以“说一句话就能启动流程”背后，实际上是一套事件驱动机制在工作。  
如果你要临时关闭全部 Hook，也可以通过环境变量（当前兼容名）`DISABLE_OMC=1` 来实现。

### 7）代码智能：不只“读文本”，而是“读结构”

OMQ 通过 MCP 接入了接近 IDE 的代码能力：查定义、查引用、做项目级诊断，以及基于 AST 的结构化搜索和替换。

最直观的例子是重命名。纯正则很容易误伤注释和同名字符串；结构化改写会更精准，只改语义上正确的位置。这类能力叠加起来，能显著减少“改了一处，漏了一片”的返工。

### 8）技能系统：把高频做法沉淀下来

技能可以理解成“可组合的工作方式”，不是换一个新助手。  
比如你希望“并行推进 + 正常实现 + 过程可追踪”，就可以通过技能组合一次性表达出来。

而且技能有项目级和用户级两种作用域：前者适合团队复用，后者适合个人习惯。你在会话里摸索出的高效套路，也可以用 `/skillify` 固化下来，之后直接复用。

---

## 四、总结

OMQ 的价值不在“让 AI 写更多代码”，而在“让 AI 以更像工程团队的方式交付结果”。

如果要用四句话概括：

- 把角色分清，减少一个人包办带来的盲区；
- 按任务选流程，不再所有任务都一个打法；
- 用验证兜底，避免“看起来完成”；
- 让关键记忆跨会话延续，少做重复沟通。
